在备胎笔记本电脑上安装ESXi

Posted on 2021-08-09 Edited on 2021-08-11

前言：该方案实际失败了，VM的网卡速度只有1Mbps，没有修好

自定义ISO，注入网卡驱动

下载VMWare的Offline Buldle

官网需要繁琐的注册流程，百度上找了个然后校验哈希

下载第三方插件

  1. 下载地址https://www.v-front.de/p/esxi-customizer-ps.html
  2. 下载后放在E:\ESXi\

下载驱动

  1. https://vibsdepot.v-front.de/wiki/index.php/List_of_currently_available_ESXi_packages
  2. 在待安装ESXi的机器上，打开设备管理器，找到对应的网卡，找到硬件ID，我这里找到的是10EC:8168，根据硬件ID，在步骤1中的网站查找网卡驱动Net55-r8168，下载VIB文件
  3. 下载后放在E:\ESXi\pkg

离线安装VMware.PowerCLI

  1. 打开https://code.vmware.com/web/tool/12.3.0/vmware-powercli，下载VMware-PowerCLI-12.3.0-17860403.zip
  2. 下载后解压放在C:\Windows\System32\WindowsPowerShell\v1.0\Modules
  3. Set-ExecutionPolicy -ExecutionPolicy RemoteSigned
  4. Import-Module VMware.PowerCLI

生成自定义ISO

  1. Set-ExecutionPolicy Unrestricted
  2. .\ESXi-Customizer-PS-v2.6.0.ps1 -izip .\ESXi670-201912001.zip -pkgDir E:\ESXi\pkg

安装ISO

  1. 使用Rufus烧录ISO，如果使用UltraISO会提示not a COM32R image
  2. 安装前记得在BIOS中开启Intel VT
  3. 按照提示安装即可

其他问题

  1. 安装后发现上传镜像/VM奇慢无比，经查看，物理网卡的网速自动协商成了10M

解决方案：

  • 将物理网卡vmnic0（驱动程序r8168）修改为100MBps全双工，虽然当时断网了，但是重启后正常，但是实测VM的速度只有1MBps（甚至10M都达不到），Esxi中的各种配置都配过了，不知道原因在哪儿，只能暂时放弃此方案

VMWare串口调试

Posted on 2021-08-03

宿主机（Windows）和虚拟机（Linux）之间通信

配置方法

  1. VMWare虚拟机关机，添加一个串口设备，选择使用命名的管道\\.\pipe\com_1，该端是服务器，另一端是虚拟机，开启轮询时主动放弃，配置完成后开机
  2. 本地打开Putty，选择Serial，Line\\.\pipe\com_1，Speed9600
  3. 宿主机->虚拟机：虚拟机内cat /dev/ttyS1，宿主机使用Putty发送命令
  4. 虚拟机->宿主机：虚拟机内echo "1" >> dev/ttyS1

如发现Putty本地无法输入，配置方法：

  • Terminal标签页勾选Implicit CR in every LF，Implici LF in every CR，Local echo修改为Force on，Local line editing 修改为Force on

准备工作

swapoff -a，vim /etc/fstab把swap一行注释掉
vim /etc/selinux/config关闭SELinux

kubeadm init的时候，CoreDNS下载不下来

解决方案，参考：https://blog.csdn.net/a749227859/article/details/118732605

把cgroups替换为systemd

然后systemctl restart docker

错误 [ERROR FileContent–proc-sys-net-bridge-bridge-nf-call-iptables]: /proc/sys/net/bridge/bridge-nf-call-iptables contents are not set to 1

解决方案：
echo "1" >/proc/sys/net/bridge/bridge-nf-call-iptables

kubeadm加入节点的命令忘了，重新查询

kubeadm token create --print-join-command

重启之坑

重启后master的Status为NotReady

使用kubectl get pod -n kube-system命令查看，发现coredns开头的一直处于Pending状态，解决方案

  1. kubectl apply -f https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml
  2. systemctl restart kubelet，重启后等待，重新拉一下pod，显示kube-flannel-ds-mgh4s这个POD的状态为Init:ErrImagePull

意思是拉取镜像失败了，应该是墙的问题，解决方案：

  1. 查看https://raw.githubusercontent.com/coreos/flannel/master/Documentation/kube-flannel.yml
  2. 搜索quay.io/coreos/flannel版本为V0.14.0，然后去https://github.com/flannel-io/flannel/releases下载对应版本的镜像（docker格式，不是tar.gz格式）。
  3. docker load < flanneld-v0.14.0-amd64.docker
  4. systemctl restart kubelet
